# AGENTS.md

## Mission

This workspace is for migrating the current Rust `box-winder` backend architecture into a TypeScript monorepo.

The existing frontend project at:

```text
/Users/chao/Downloads/awsing
```

should be reused as the web frontend. The migration focus is the backend/control-plane side:

- Reimplement the current Rust backend logic in TypeScript.
- Use NestJS for the backend API and daemon-oriented service code.
- Preserve the current architecture principles.
- Turn the result into a monorepo containing frontend, backend API, daemon worker, CLI, shared core logic, and database code.

This migration is not only a language rewrite. It is an architecture migration.

## Target Monorepo Shape

Use `awsing` as the frontend application and migrate it into the monorepo structure.

Recommended target layout:

```text
repo/
├─ apps/
│  ├─ web/        # existing awsing Next.js frontend
│  ├─ api/        # NestJS backend API
│  ├─ daemon/     # NestJS/TypeScript worker process
│  └─ cli/        # TypeScript CLI
├─ packages/
│  ├─ core/       # shared pure business logic
│  ├─ db/         # schema, migrations, queries
│  ├─ config/     # shared environment/config loading
│  └─ contracts/  # shared API types/schemas
└─ AGENTS.md
```

If moving files is needed, preserve the existing frontend behavior first, then refactor paths incrementally.

## Product Boundaries

The frontend and backend may use different product names.

Recommended model:

```text
Frontend product name: awsing or a user-facing console brand
Backend/control-plane name: box-winder
API service: box-winder-api
Daemon service: box-winder-daemon
CLI command: box-winder
```

Frontend branding can change independently. Backend service names, CLI command names, database schema names, Docker image names, and systemd unit names should remain stable once chosen.

## Architecture Direction

The target system is an event-driven sing-box control plane.

Target data flow:

```text
Frontend / CLI
-> NestJS API or approved DB function
-> PostgreSQL users/events
-> daemon worker
-> sing-box config + reload
```

The frontend expresses intent. The API validates and records facts. PostgreSQL stores durable facts and events. The daemon worker performs external side effects.

## Managed Sing-Box Ownership

`box-winder` should own and manage its own sing-box runtime surface.

`box-winder` owns:

- its PostgreSQL schema
- its sing-box config templates
- its rendered sing-box config
- its managed sing-box process
- its logs, runtime files, and sync status

`box-winder` does not own:

- the user's existing sing-box config
- the system's existing sing-box service
- unrelated sing-box processes

The daemon worker should render the box-winder-owned sing-box config and manage only the box-winder-owned sing-box process. Existing system sing-box installations may continue to run independently.

Do not design the migrated system around mutating `~/.config/sing-box/config.json` or another pre-existing user/system config. The box-winder-managed config should live in an explicit owned path, such as `/var/lib/box-winder/sing-box/config.json` for self-hosted deployments.

## Port Cluster Rules

Use a configurable base port to derive all managed proxy ports. The default base port should be:

```text
SING_BOX_PORT_BASE=10000
```

Protocol ports should be generated by stable offsets from the base port instead of being hardcoded independently.

Recommended default offset map:

```text
shadowsocks = SING_BOX_PORT_BASE + 0
vmess       = SING_BOX_PORT_BASE + 1
trojan      = SING_BOX_PORT_BASE + 2
naive       = SING_BOX_PORT_BASE + 3
hysteria2   = SING_BOX_PORT_BASE + 4
tuic        = SING_BOX_PORT_BASE + 5
anytls      = SING_BOX_PORT_BASE + 6
shadowtls   = SING_BOX_PORT_BASE + 7
```

Server config rendering and subscription rendering must use the same port resolver from shared core logic.

The implementation should validate that generated ports are in the valid TCP/UDP port range and should check whether the full managed port cluster is available before starting or reloading the box-winder-managed sing-box process.

Changing `SING_BOX_PORT_BASE` should move the whole managed port cluster together. This makes it easier to run box-winder alongside an existing system sing-box instance or to run multiple isolated box-winder deployments on the same host.

## Source Of Truth

PostgreSQL is the source of truth for durable state.

PostgreSQL owns:

- users
- roles
- password hashes
- subscription tokens or token metadata
- invite codes
- enabled/disabled/deleted state
- outbox events
- sync status
- admin/audit logs
- usage snapshots if added later

Derived artifacts must be rebuildable from PostgreSQL state plus static templates/configuration.

Derived artifacts include:

- sing-box server config
- per-user subscriptions
- generated protocol credentials
- rendered config snapshots
- sync status output

The database records facts and events only. It must not write files or reload services.

## Backend API Rules

Use NestJS for the backend API.

The API is responsible for:

- authentication
- authorization
- admin permission checks
- request validation
- database transactions
- durable event creation
- returning operation status

The API should write facts, not perform long-running sync side effects.

Avoid designs where an HTTP request handler directly writes sing-box config or reloads sing-box. If temporary compatibility requires this during migration, mark it clearly as a migration bridge and remove it after the daemon flow works.

## Authentication And Session Rules

Use NestJS JWT and Passport libraries for authentication.

Required libraries:

- `@nestjs/passport`
- `passport`
- `passport-jwt`
- `@nestjs/jwt`

Login should issue a JWT for frontend, CLI, and API access. JWT validation should use Passport strategies and NestJS guards.

Session persistence is required. The backend must record login sessions in PostgreSQL so sessions can be queried, revoked, expired, and audited. JWT verification may remain stateless, but authorization must also respect persisted session state when a session identifier is present.

JWT payloads should include at least:

- user id
- role
- session id
- issued-at / expiry information handled through standard JWT claims

The API must support logout/session revocation. Admin-facing APIs may later expose active session lists and forced revocation.

JWT expiration must be configurable through environment variables. Use:

```text
JWT_EXPIRES_IN=8h
```

Default JWT expiration is `8h` when the environment variable is not set.

Authorization remains an API-layer responsibility. Workers should not make user-intent authorization decisions.

## Daemon Worker Rules

Use a separate TypeScript/NestJS worker process for daemon responsibilities.

The daemon worker is responsible for:

- consuming durable database events
- rendering sing-box config from enabled, non-deleted users
- writing config atomically
- starting, stopping, reloading, or restarting the box-winder-managed sing-box process
- recording sync success/failure
- retrying failed sync work safely
- exposing enough logs/status for admin visibility

The worker should prefer full rendering from enabled, non-deleted users instead of patching individual users into config. This avoids stale partial updates and event ordering problems.

The worker should not make authorization decisions for user intent. Authorization belongs in the API layer.

The worker must never start, stop, reload, restart, or mutate unrelated sing-box processes.

## Frontend Rules

Reuse the existing `awsing` Next.js frontend.

The frontend should:

- call the backend API
- use shared API contract types where practical
- display backend-reported status
- never connect directly to PostgreSQL
- never write sing-box config
- never reload or restart sing-box
- support separate deployment from the backend, such as Vercel frontend plus self-hosted backend

If the frontend uses Next.js route handlers, they may act as a thin proxy/BFF only. They must not become the real control plane.

## CLI Rules

Implement the CLI in TypeScript.

Default CLI behavior:

```text
CLI -> NestJS API -> PostgreSQL facts/events -> daemon sync
```

Trusted operational behavior may use approved database functions directly when explicitly designed.

The CLI should not maintain a separate configuration-sync path. It should share core logic through packages and rely on the daemon worker for external side effects.

Optional `--wait` behavior may wait for worker sync completion.

## Core Package Rules

Put pure shared logic in `packages/core`.

Core logic should include:

- token generation
- password hashing helpers
- credential derivation
- supported protocol definitions
- sing-box config rendering
- subscription rendering
- node naming helpers
- validation primitives that are not framework-specific

Avoid duplicating credential or rendering logic between API, daemon, CLI, and frontend.

## Database Package Rules

Put schema, migrations, and database helpers in `packages/db`.

The schema should support:

- users with roles
- invites
- durable outbox/events
- sync status
- audit/activity logs
- persistent sessions
- session revocation and expiration metadata

Use an outbox-style pattern so worker sync survives API crashes, worker restarts, and process failures.

Database migrations must be explicit and reviewable.

## Existing Rust Behavior Reference

The current Rust project remains the behavioral reference during migration.

Important behavior to preserve unless intentionally changed:

- user registration flow
- invite-code flow
- login behavior
- token generation prefix behavior
- credential derivation
- enabled-user filtering
- subscription rendering formats
- sing-box config rendering
- atomic config writes
- CLI operational flows

When the TypeScript implementation intentionally differs from Rust behavior, document the difference.

## Migration Strategy

Prefer incremental migration.

Recommended order:

1. Move or wrap the existing `awsing` frontend into `apps/web`.
2. Create monorepo tooling without changing frontend behavior.
3. Port pure Rust core logic into `packages/core`.
4. Add tests for token, credential, subscription, and config rendering parity.
5. Create `packages/db` with PostgreSQL schema and migrations.
6. Implement NestJS API in `apps/api`.
7. Implement outbox/event creation for user-changing operations.
8. Implement daemon worker in `apps/daemon`.
9. Implement TypeScript CLI in `apps/cli`.
10. Wire frontend to the new API.
11. Add deployment docs for Vercel frontend and self-hosted backend/daemon.
12. Remove temporary bridges after parity is verified.

Do not rewrite frontend UI while migrating backend architecture unless needed for API integration.

## Deployment Model

Support separated deployment.

Target deployment:

```text
Vercel or web host
-> apps/web

Self-hosted server
-> apps/api
-> apps/daemon
-> PostgreSQL
-> box-winder-managed sing-box
```

The daemon must run on the server with permission to write the box-winder-owned sing-box config and start, stop, reload, or restart the box-winder-managed sing-box process.

Recommended self-hosted owned paths include `/var/lib/box-winder/sing-box/config.json`, `/var/log/box-winder`, and `/run/box-winder`.

The frontend should use a configurable API base URL.

## Tooling Preferences

Prefer the existing frontend tool choices where possible:

- TypeScript
- Bun
- Next.js for frontend
- NestJS for backend API and worker structure
- PostgreSQL
- Biome if already adopted by the frontend

Avoid introducing multiple competing tools for the same responsibility.

Keep `mise` as the project configuration and environment-variable management entrypoint.

The migrated monorepo should include `mise.toml` for local development tasks and shared environment defaults. Environment variables should be documented there or in a linked environment reference.

JWT expiration should be configured through `JWT_EXPIRES_IN`, defaulting to `8h`.

`SING_BOX_PORT_BASE` should default to `10000` and be managed through `mise` for local development.

## Testing Expectations

Tests should cover:

- token generation
- credential derivation stability
- subscription rendering
- sing-box config rendering
- API validation and authorization
- database transactions and event creation
- daemon worker idempotency and retry behavior
- CLI calls to API
- frontend API integration states
- login issues JWT and persists a session
- JWT expiration respects `JWT_EXPIRES_IN` and defaults to `8h`
- revoked sessions cannot access protected APIs
- role checks reject non-admin users for admin APIs
- port cluster generation uses stable protocol offsets from `SING_BOX_PORT_BASE`
- generated ports are validated before config render or process reload
- server config and subscription rendering resolve ports through the same shared function
- daemon process management is isolated to the box-winder-managed sing-box instance

Rendering tests should use stable fixtures.

## Non-Goals

Do not:

- make the frontend the source of truth
- let Vercel/serverless code perform daemon work
- bypass PostgreSQL for durable user state
- keep a separate CLI-only sync implementation
- store derived sing-box config as authoritative state
- duplicate credential derivation across apps
- mix frontend branding with backend service identity
- modify user/system existing sing-box config
- manage unrelated sing-box processes
